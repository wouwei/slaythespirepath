Rethinking...

I think THIS is an important step, when programming . The quicker your do it the better it is: as the algo went too complex , I decided to rethink it .. Plus the first demand was to manage xroads , not everything else. So this time , in the limit of the grid :
  • we randomly generate on which step the xroads pops for all the pathes . those xroads must follow a '' gap '' rule , which means, must have a minimum distance between them.
  • we randomly generate pathes from the left to the right.
  • each path is generated inside a ''corridor ''. the corridor play the limiter
  • at first we divide the grid on equal size corridors.
  • Which means, the right border of each corridor really depends on the columns ( or width ) of the grid , and the number of pathes.
  • as for the left limit, we will base it on the already generated path on the left ... And the shared xroad with it...
  • while randomly generating the path inside the pre-determined corridor, we take into account some esthetic constraint with the xroads with the neighbour path on the right :
      if the next step is an xroad, the path cannot go on the left. Else the right path will take the same '' way '' of the path, in order to reach the xroad.
      if we are on the xroad, the path cannot go on the rifht . Else the right path can only take the same '' way '' of the path.
      if we are on the xroad , and the right limit of the corridor go on the left, the path cannot go straight, too....


As the algorithm goes, there are still some irregularities, like the corridor constraints are so that two pathes can follow the same way for a long time, even if we avoid it ,or a path can suddendtly ''jump'' 2 or 3 cases on the left or the right, even if we try to move by one case only... but as 90% of the time the algo works, we can still reroll. Correcting it will add some more complexity to the algorithm, which i do not intend to do.
